# System Promises

| Promise | Source | User/operator impact | Confidence |
|---|---|---|---|
| Agave can be built and tested from source as a single workspace, but the default debug build is not considered production-safe for validators. | `README.md`, `Cargo.toml` | Operators and auditors must distinguish developer builds from production deployment artifacts. | High |
| A consensus validator and a full RPC node are different operating modes with different resource and safety expectations; a full RPC node typically should not also be the active voting node. | `docs/src/operations/validator-or-rpc-node.md`, `docs/src/operations/setup-an-rpc-node.md` | Role separation is part of maintaining node health and avoiding self-inflicted overload. | High |
| Tower BFT is intended to make rollback cost grow with confirmations, so validators converge on heavier forks while making inconsistent voting increasingly expensive. | `docs/src/implemented-proposals/tower-bft.md` | This is a core client-visible safety/finality promise. | Medium |
| Vote propagation is designed so votes are eventually delivered and eventually encoded into the ledger even if the current leader is unavailable or flooded. | `docs/src/implemented-proposals/reliable-vote-transmission.md` | Liveness depends on asynchronous propagation rather than a single leader receiving direct UDP traffic. | Medium |
| Repair is a best-effort recovery mechanism for missing shreds and orphan slots, constrained to data inside the current verifiable epoch. | `docs/src/implemented-proposals/repair-service.md` | Replay catch-up is not guaranteed by first-pass delivery alone; repair is an explicit degraded-mode path. | Medium |
| Durable transaction nonces let clients use longer-lived transactions without making the nonce reusable, including after some failure cases. | `docs/src/implemented-proposals/durable-tx-nonces.md` | Important anti-replay guarantee for offline or slow-signing workflows. | High |
| Validator bootstrap should be constrained to trusted peers when possible: `--known-validator` and `--only-known-rpc` are recommended to reduce malicious snapshot and genesis download risk. | `docs/src/operations/guides/validator-start.md`, `docs/src/operations/setup-an-rpc-node.md` | Trust in bootstrap peers is an explicit part of the safety model. | High |
| Snapshot correctness is intended to be verifiable against voted account-hash state, but current security docs still treat malicious snapshots as effectively trust-on-first-use with known shortcomings. | `docs/src/implemented-proposals/snapshot-verification.md`, `SECURITY.md` | Snapshot trust is a material reliability boundary and remains ambiguous enough to verify in code later. | Low |
| The built-in RPC service is not intended to be exposed directly to untrusted clients; safe operation assumes a reverse proxy or equivalent protections. | `docs/src/operations/setup-an-rpc-node.md`, `SECURITY.md` | RPC availability and abuse-resistance are partly delegated to deployment architecture. | High |
| Local ledger retention is intentionally bounded by `--limit-ledger-size`, which means complete historical RPC service may require an external archive such as Bigtable. | `docs/src/operations/guides/validator-start.md`, `docs/src/operations/setup-an-rpc-node.md` | History completeness is conditional, not absolute. | High |
| Validator failover can be performed with virtually no downtime and without safety issues if the operator preserves tower/identity continuity and uses `--require-tower` during takeover. | `docs/src/operations/guides/validator-failover.md` | Recovery correctness depends on strict operational choreography, not just process restart. | Medium |
| Withdrawer authority must remain off-validator and on a trusted machine; compromise or loss of that key changes who ultimately controls vote-account funds and authority. | `docs/src/operations/setup-a-validator.md`, `docs/src/operations/best-practices/security.md` | Human key-handling is a first-class system invariant. | High |
| Some extensions are outside the validator's defended reliability boundary: non-conforming Geyser plugins and `scheduler-bindings` processes are treated as operator-trusted integrations. | `SECURITY.md` | Certain crashes or instability modes are accepted if operators attach untrusted extensions. | High |
